## 1. Functional Requirements

### REQ-1: Email OTRS ticket ID tracked in local DB

### REQ-2: Pre-check before ticket creation — skip if ticket already exists

In `ActionExecutor._create_ticket()`, **before** calling `TicketIngestionService.ingest_email()`, query the `Email` record:

```
1. Look up `Email` by message_id from `ctx.raw.message_id`
   (already auto-generated by _save_email() if email had none — see REQ-4)
2. If `email.otrs_ticket_id` is not None:
   → Log info "Ticket {id} ya existe para email {message_id}"
   → Return ActionResult(success=True, detail=f"Ticket {email.otrs_ticket_id} ya existe")
3. If email record not found (shouldn't happen — _save_to_db() always runs first):
   → Log warning, proceed to create ticket (best-effort, no post-save possible)
```

The lookup key is **`message_id`** because:
- It is the stable identifier that connects the `EmailContext` (pipeline) with the `Email` record (DB).
- `_save_email()` already ensures every email has a message_id — either from the original email header or an auto-generated UUID (see REQ-4).
- The `Email` model has `UniqueConstraint("message_id", "account_id")`, guaranteeing at most one match.

---

### REQ-3: Post-save after ticket creation — store ticket ID

After `service.ingest_email()` succeeds, update the `Email` record:

```
1. Set email.otrs_ticket_id = ticket.id (the ID returned from OTRS)
2. Set email.otrs_ticket_created_at = datetime.now(timezone.utc)
3. await self.db.commit()
```

**Error handling**:
- If the post-save commit fails → log error (OTRS ticket already exists, orphan risk is acceptable for MVP — see proposal risks).
- If `ingest_email()` raises → the `except` block in `_create_ticket()` already catches it; no otrs_ticket_id is saved → next retry will attempt a new ticket creation (scenario 3).

**Why not use `db.flush()`**: The post-save must be committed immediately so that subsequent pipeline runs for the same email see the saved `otrs_ticket_id`. A flush alone wouldn't persist across sessions.

### REQ-4: Auto-generated message_id fallback if email has none

**Already implemented** in `ActionExecutor._save_email()` (lines 248–251):

```python
message_id = ctx.raw.message_id
if not message_id:
    message_id = f"auto-{uuid.uuid4()}"
```

This guarantees every `Email` record has a `message_id`, which is the lookup key for REQ-2.

**Implication for REQ-2**: Because `_save_email()` runs before `_create_ticket()` (step 1 vs step 6 in `execute_all()`), the `message_id` in `ctx.raw.message_id` is **not** updated with the auto-generated value. The pre-check query in REQ-2 must use the `message_id` value that was actually stored in the DB, not the possibly-None value from `ctx.raw`.

**Implementation note**: Either:
- **(A)** Update `ctx.raw.message_id` after auto-generation in `_save_email()` (side effect on the context), OR
- **(B)** Query by both `message_id` from ctx and auto-generated pattern as fallback:
  ```python
  # B1: try with ctx's message_id
  if ctx.raw.message_id:
      result = await self.db.execute(
          select(Email).where(Email.message_id == ctx.raw.message_id)
      )
      email = result.scalar_one_or_none()
  # B2: if not found, fall back to most recent email from same sender
  if not email:
      result = await self.db.execute(
          select(Email).where(Email.sender_email == ctx.raw.sender_email)
          .order_by(Email.created_at.desc())
      )
      email = result.scalar_one_or_none()
  ```
  Option **(A)** is simpler and recommended. Update `_save_email()` to set `ctx.raw.message_id = message_id` after auto-generation. This is a minimal, local change.

## 2. Scenarios (Gherkin)

### Scenario 1: Same email processed twice → second pass skips ticket creation

```
Background:
  Given OTRS is configured and reachable
  And the pipeline processed an email with message_id "msg-001"
  And a ticket "T-100" was created in OTRS
  And Email.otrs_ticket_id was set to "T-100"

Scenario: Second pipeline run for the same email skips OTRS
  When the same email with message_id "msg-001" enters the pipeline again
  And _save_to_db() detects duplicate message_id and returns the existing Email record
  And _create_ticket() queries Email by message_id "msg-001"
  Then Email.otrs_ticket_id is "T-100"
  And _create_ticket() does NOT call TicketIngestionService.ingest_email()
  And the returned ActionResult has:
    - action: "otrs_ticket_create"
    - success: True
    - detail: "Ticket T-100 ya existe"
```

**Implementation verification**:
- Assert `ingest_email()` mock is NOT called on second pass.
- Assert `ActionResult.detail` contains `"ya existe"`.

---

### Scenario 2: Email without message_id → auto-generates UUID, dedup still works

```
Background:
  Given an email arrives WITHOUT a Message-ID header
  And the pipeline runs

Scenario: Auto-generated message_id enables dedup
  When _save_email() detects ctx.raw.message_id is None
  Then it generates "auto-<uuid>" as the message_id
  And the Email record is saved with that auto-generated message_id
  And the Email record's otrs_ticket_id is set after ticket creation
  When the same email enters the pipeline again (e.g. IMAP re-fetch)
  And _save_email() detects duplicate by message_id and returns existing Email
  And _create_ticket() queries Email by the auto-generated message_id
  Then Email.otrs_ticket_id is NOT None
  And ticket creation is skipped
```

**Implementation verification**:
- Assert `Email.message_id` starts with `"auto-"`.
- Assert second pipeline pass does not create a second OTRS ticket.

---

### Scenario 3: OTRS ticket creation fails → no otrs_ticket_id saved, next retry creates new ticket

```
Background:
  Given OTRS is configured
  And an email with message_id "msg-002" exists in the DB
  And Email.otrs_ticket_id is NULL

Scenario: OTRS failure does NOT leave a stale otrs_ticket_id
  When _create_ticket() calls TicketIngestionService.ingest_email()
  And ingest_email() raises an exception (e.g. network error, OTRS 500)
  Then the exception is caught by the existing try/except block
  And Email.otrs_ticket_id remains NULL
  And Email.otrs_ticket_created_at remains NULL
  And no commit is made for the otrs_ticket_id fields
  And ActionResult(success=False, detail="<error>") is returned
  When the same email is processed again (retry)
  Then _create_ticket() finds otrs_ticket_id is NULL
  And it proceeds to call ingest_email() again
  And a NEW ticket is created in OTRS
```

**Implementation verification**:
- Assert `Email.otrs_ticket_id` is NULL after `ingest_email()` raises.
- Assert `ingest_email()` mock is called again on retry.

### Scenario 4: Different emails → both get their own tickets independently

```
Background:
  Given two distinct emails with message_ids "msg-A" and "msg-B"
  And no otrs_ticket_id exists for either

Scenario: Independent ticket creation for different emails
  When msg-A is processed
  Then a ticket is created in OTRS
  And Email(msg-A).otrs_ticket_id is set to ticket ID "T-A"
  When msg-B is processed
  Then a ticket is created in OTRS
  And Email(msg-B).otrs_ticket_id is set to ticket ID "T-B"
  And both emails have different otrs_ticket_id values
```

**Implementation verification**:
- Assert two distinct OTRS tickets were created.
- Assert each Email record has its correct ticket ID.

## 3. Non-functional Requirements

### NFR-1: Zero impact on pipeline when no otrs_ticket_id exists (new emails)

The pre-check (REQ-2) must be a **cheap SELECT** that adds no measurable overhead:

```python
# Pre-check (single query by PK-like field)
result = await self.db.execute(
    select(Email).where(Email.message_id == message_id)
)
email = result.scalar_one_or_none()
if email and email.otrs_ticket_id:
    return ActionResult(...)  # skip
```

- No extra network calls.
- No locks or transactions beyond the existing session.
- No new indexes needed (already covered by `uq_message_id_account` unique constraint, which is indexed automatically).

### NFR-2: Must not break existing tests

- All existing test cases for `_create_ticket()` must pass without modification.
- The pre-check is a no-op when `Email.otrs_ticket_id` is NULL (which is the case for all existing test data).
- New tests are additive — see [tests/ directory] for the new test module.

---

## 4. Out of Scope

- **Content-fingerprint dedup** (hash-based): Not needed for MVP. Only message_id-based dedup.
- **OTRS-side search dedup**: No OTRS search API call to check for existing tickets before creation. Pure local-DB approach.
- **Retroactive migration** of existing tickets: Only affects new pipeline runs. Existing Email records with no `otrs_ticket_id` will simply have NULL and get tickets created on their next pipeline run.
- **Cross-account dedup**: Scoped to same-email dedup only (message_id + account_id).
- **Cascade deletion** of OTRS tickets when an Email is deleted: Out of scope. OTRS tickets are owned by OTRS, not by the local DB.
